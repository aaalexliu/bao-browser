The bet

The LLM is not the runtime.

Agent tools ask a model to re-decide every click, every run - so they're slow, metered, and only mostly right. Bao uses a model once, at authoring time, to turn your recording into readable steps. Replay after that is ordinary code.

Bao Deterministic replay

  • ✓Free & instant. No tokens per run. Replay is code, not a prompt.
  • ✓Same result every time. A ranked selector ladder re-finds each element - no dice roll.
  • ✓Offline. Runs entirely in the browser, on any page you choose.
  • ✓Fails cleanly. When a page truly changed, you get “re-record this step,” not a wrong click.

LLM agents Probabilistic each run

  • ✗Metered - every run spends tokens and wall-clock latency.
  • ✗Non-repeatable - the model may take a different path on identical input.
  • ✗Needs the network and a live model to do anything at all.
  • ✗Fails silently - a confident wrong action looks just like a right one.

How it works

Three steps, and the middle one is the whole idea.

01 · RECORD

Do the task once

Open the side panel, hit record, and click through your workflow. Steps appear live as you go - assertions too, if you want replay to check the page.

02 · COMPILE

Get readable, editable steps

Your recording becomes a clean workflow you can actually read - reorder, delete, or edit a value with no re-recording. Sensitive fields like passwords are masked at capture.

03 · REPLAY

Run it deterministically

Replay in the browser with live ✓/✗ progress, or headless in CI. Export the workflow as JSON and drive it from a one-line test runner.

Why replay holds up

A resolution ladder, not a guess.

For every step, Bao walks the cheapest deterministic way to re-find your element and stops at the first that works. It escalates per step, never globally.

TierHow it re-finds the elementDeterminismStatus
0Unique robust selector - test-id, id, or ariafullshipped
1Content anchor + within-descriptor, across navigationsfullshipped
2DOM subtree + screenshot - audit fuel, not a resolvern/ashipped
3VLM self-heal on a miss → writes a fresh selector backre-cachednext
4CDP executor - trusted input, file upload, closed shadowfullopt-in

Any escalation is cached back down to a deterministic selector, so the expensive tier runs at most once. Light-DOM, single-origin tasks live at tiers 0-1.

For QA & CI

Record a test. Run it headless. Get a clean ✓/✗.

The engine is the product here - no backend required. Capture assertions while you record, export the workflow as JSON, and replay it from a thin runner that exits non-zero when a step or assert fails.

bao — test runner
$ node test/run.mjs weekly-report.json https://app.acme.com

replaying 6 steps + 2 assertions…

  ✓ 01  go to /login              via nav
  ✓ 02  type into “Email”         via aria
  ✓ 03  password (you enter it)   masked
  ✓ 04  click “Sign in”           via testid
  ✓ 05  open Reports → Weekly     via anchor
  ✓ 06  download report.csv       via download
  ✓ assert  “Q3 Summary” present
  ✗ assert  row count = 42        got 41

✗ 1 assertion failed  — exit 1
  • assertionsRecord what to check, not just what to do. textPresent, elementVisible/absent, urlMatches - non-fatal, all reported, the run fails if any fails.
  • headlessOne command, honest exit code. node test/run.mjs steps.json url - exit 0 all-green, 1 on any failure. Drop it straight into CI.
  • portableThe workflow is just JSON. Export from the panel or dashboard; it's directly consumable by the runner - no proprietary format, readable in a diff.
  • stableSurvives a churning feed. Anchored capture re-resolves the same card by its stable id even after the list reorders.
